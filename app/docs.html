VSCode Session Chat Acquisition & Parsing — How Data Is Parsed & Normalized

VS-CAP is a forensic toolkit for collecting and analyzing VS Code workspace data, with a focus on AI/Copilot chat interactions. It consists of three components:

  1. KAPE Target (VSCode_WorkspaceStorage.tkape) — Forensically sound collection of VS Code artifacts
  2. VS-CAP Viewer (vs-cap-viewer.html) — Browser-based evidence parser and viewer
  3. This Documentation — Technical reference for data formats and parsing methodology
🔒 Privacy by Design The VS-CAP Viewer runs entirely in the browser. No data is uploaded to any server. All evidence data is held in JavaScript memory and is completely purged when the page is refreshed or the "Clear Data" button is clicked.

KAPE Collection

The VSCode_WorkspaceStorage.tkape target collects the following artifacts:

TargetPathDescription
workspaceStorage...\User\workspaceStorage\All workspace-specific data (chat sessions, edit sessions, Copilot artifacts, workspace state databases)
globalStorage...\User\globalStorage\Global extension data (cross-workspace preferences, extension state)
settings.json...\User\settings.jsonUser VS Code configuration and preferences
profiles...\User\profiles\VS Code user profiles (if multiple profiles are configured)

The %user% variable ensures collection from all user profiles on the target system. Both VS Code Stable and Insiders paths are included. The RecreateDirectories: true flag preserves the original folder hierarchy in the output.

⚠️ Collection Note Run KAPE as SYSTEM (not just Administrator) for the fewest file access errors. Use: PsExec64.exe -s kape.exe [options]

workspace.json — Workspace Identity Mapping

Each hash-named folder in workspaceStorage/ corresponds to a VS Code workspace (folder or project). The workspace.json file inside each hash folder maps the hash to the actual workspace path.

Structure

{
  "folder": "file:///d%3A/%21projects/my-project"
}

Fields

FieldTypeDescription
folderString (URI)URI-encoded path to the workspace folder. Decode with decodeURIComponent() after stripping the file:/// prefix.
workspaceString (URI)Alternative field used for multi-root workspaces (.code-workspace files).

Parsing Logic

// VS-CAP decodes the folder URI to a human-readable path:
const raw = wsJson.folder;   // "file:///d%3A/%21projects/my-project"
const decoded = decodeURIComponent(raw.replace(/^file:\/\/\//i, ''));
// Result: "d:/!projects/my-project"
// Short name (last path segment): "my-project"

The hash folder name itself is an opaque identifier. The workspace.json mapping is the only reliable way to associate a hash with a project.

Chat Sessions — JSONL Format

Chat sessions are stored in chatSessions/{uuid}.jsonl files using the JSON Lines format (one JSON object per line). Each file represents one complete chat conversation with an AI assistant (GitHub Copilot).

File Location

workspaceStorage/{hash}/chatSessions/{session-uuid}.jsonl

Line Types

Each line in the JSONL file is a JSON object with a kind field indicating its type:

KindNamePurpose
Kind 0 Initial State Full snapshot of the session state at time of first save. Contains all metadata, model info, and may include already-completed requests.
Kind 1 Scalar Mutation Updates a single value at a specified path in the session state. Used for setting titles, updating input text, setting results, etc.
Kind 2 Array Mutation Replaces an array or array element at a specified path. Used for adding/updating requests and their response arrays.

JSONL Mutation Model — State Reconstruction

VS Code uses an incremental mutation model to efficiently serialize chat session state. Rather than saving the full state after every change, it records the initial state (Kind 0) followed by a series of patch operations (Kind 1/2).

Kind 0 — Initial State

{
  "kind": 0,
  "v": {
    "version": 3,
    "creationDate": 1773609865809,
    "customTitle": "My Chat Session",
    "sessionId": "797b631e-e462-4fb9-86ef-1881071066c2",
    "responderUsername": "GitHub Copilot",
    "requests": [],
    "inputState": {
      "selectedModel": {
        "identifier": "copilot/claude-opus-4.6",
        "metadata": { "name": "Claude Opus 4.6", ... }
      },
      "mode": { "kind": "agent", "id": "agent" }
    }
  }
}

The v property contains the complete session state object. If the session already had exchanges at save time, the requests array will be pre-populated.

Kind 1 — Path-Based Mutation

// Set the session title:
{"kind":1,"k":["customTitle"],"v":"Docker Web UI Control Script"}

// Set the user's input text (what they typed):
{"kind":1,"k":["inputState","inputText"],"v":"Create a PS1 script to manage Docker containers"}

// Set the result of request index 0:
{"kind":1,"k":["requests",0,"result"],"v":{"timings":{"totalElapsed":85251},...}}
  • k — Path array. Each element is a key (string) or index (number) to navigate the state object.
  • v — New value to set at the specified path.

Kind 2 — Array Replacement

// Replace the entire requests array (new request added):
{"kind":2,"k":["requests"],"v":[{ "requestId": "request_xxx", "timestamp": 1773609980479, ... }]}

// Replace the response array of request 0 (AI response updated):
{"kind":2,"k":["requests",0,"response"],"v":[
  {"kind":"thinking","value":"Let me analyze this..."},
  {"value":"Here's the solution..."}
]}

Reconstruction Algorithm

How VS-CAP reconstructs the full session state: 
function reconstructSession(jsonlText) {
  const lines = jsonlText.split('\n').filter(l => l.trim());
  let state = {};

  for (const line of lines) {
    const entry = JSON.parse(line);

    if (entry.kind === 0) {
      // Initial state — deep clone
      state = structuredClone(entry.v);
    } else if (entry.kind === 1 || entry.kind === 2) {
      // Navigate path and set value
      let target = state;
      const path = entry.k;
      for (let i = 0; i < path.length - 1; i++) {
        if (target[path[i]] === undefined) {
          target[path[i]] = typeof path[i+1] === 'number' ? [] : {};
        }
        target = target[path[i]];
      }
      target[path[path.length - 1]] = entry.v;
    }
  }

  return state; // Fully reconstructed session
}

The algorithm applies all mutations in order. The final state reflects the session as it existed when VS Code last persisted it. Both Kind 1 and Kind 2 use identical path-navigation mechanics; the distinction is semantic (Kind 2 indicates the value is an array).

Request & Response Structure

Each entry in the requests array represents one exchange (user prompt + AI response).

Request Object

FieldTypeDescription
requestIdStringUnique identifier for this request (e.g., request_50502a1e-...)
timestampNumberEpoch milliseconds when the request was sent
agentObjectAgent metadata — fullName, name, id, available slash commands, extension info
modelIdStringModel identifier (e.g., copilot/claude-opus-4.6)
responseArrayAI response elements (text, thinking, tool calls — see below)
resultObjectCompletion metadata — timings, metadata.renderedUserMessage
modelStateObjectCompletion state — value (0=pending, 1=complete), completedAt
contentReferencesArrayFiles/URIs referenced in the context
codeCitationsArrayCode citation information
followupsArraySuggested follow-up questions

Response Element Types

The response array contains objects of different types:

TypeIdentificationDescription
Text{ value: "..." }Markdown text content from the AI. The primary response text.
Thinking{ kind: "thinking", value: "..." }AI reasoning/chain-of-thought (when using models with extended thinking). Includes optional generatedTitle.
Thinking Done{ kind: "thinking", metadata: { vscodeReasoningDone: true } }Marker indicating thinking phase is complete. Filtered out by VS-CAP.
Tool Call{ kind: "toolInvocationSerialized" }Tool invocation — file read, terminal command, search, sub-agent execution, etc. Contains invocationMessage and pastTenseMessage.
MCP Servers{ kind: "mcpServersStarting" }MCP (Model Context Protocol) server startup events. Informational only.

GitHub Copilot Chat Artifacts

Chat Session Resources

Located at GitHub.copilot-chat/chat-session-resources/{session-uuid}/{tool-id}/content.txt

These are outputs from tool invocations during chat sessions — terminal output, file contents, search results, etc. They are linked to the parent session via the session UUID in the path.

Agent Memory

Located at GitHub.copilot-chat/memory-tool/memories/{base64-uuid}/plan.md

Persistent memory files created by the Copilot agent. These contain plans, notes, and context that the agent stores between sessions. The folder name is a Base64-encoded session UUID.

Content is standard Markdown and is rendered by VS-CAP using the marked.js library.

state.vscdb — SQLite Workspace State

Each workspace hash folder contains a state.vscdb SQLite database storing VS Code's workspace-specific state.

Database Schema

-- Primary table
CREATE TABLE ItemTable (key TEXT UNIQUE ON CONFLICT REPLACE, value BLOB);

The database uses a simple key-value store in the ItemTable. Values are typically JSON strings.

Notable Keys

Key PatternContent
workbench.panel.*Panel positions, visibility, sizes
memento/workbench.editors.*Recently opened files and editor state
workbench.explorer.*File explorer state (expanded folders, scroll position)
extensionStorage/*Extension-specific persistent data
debug.*Debug configuration state
terminal.*Terminal session state

VS-CAP uses sql.js (SQLite compiled to WebAssembly) to parse these databases entirely in the browser.

Settings & Profiles

settings.json contains the user's VS Code configuration. VS-CAP groups settings by their namespace prefix (e.g., editor.*, workbench.*, github.*) for easy browsing.

User profiles in the profiles/ directory contain profile-specific overrides.

Lazy Loading Strategy

For large datasets (100+ workspaces, many sessions), VS-CAP uses lazy loading:

  • Step 4: Only the first line (Kind 0) of each JSONL file is parsed to extract session metadata (title, date, model, request count). This is fast even for large files.
  • Step 5: Full parsing (all mutations applied) only occurs when the user opens a specific session. Parsed results are cached in memory.
  • This approach enables handling datasets with thousands of sessions without browser slowdown.

User Message Extraction

The user's message is stored in a complex format that includes system context. VS-CAP extracts the actual user request from this wrapper.

Raw Rendered Message Format

<context>
The current date is March 9, 2026.
</context>
<editorContext>
The user's current file is d:\project\app.py.
</editorContext>
<reminderInstructions>
[System instructions for the AI model]
</reminderInstructions>
<userRequest>
Create a function that handles user authentication
</userRequest>

Extraction Logic

// Primary source: result.metadata.renderedUserMessage
const fullText = request.result.metadata.renderedUserMessage
  .map(r => r.text || r.value || '').join('\n');

// Extract just the user's request:
const match = fullText.match(/<userRequest>([\s\S]*?)<\/userRequest>/i);
const userMessage = match ? match[1].trim() : fullText;

// Also extract context sections for forensic completeness:
const context = fullText.match(/<context>([\s\S]*?)<\/context>/i);
const editorCtx = fullText.match(/<editorContext>([\s\S]*?)<\/editorContext>/i);
🔍 Forensic Value The full rendered message reveals what context the AI model actually received, including system instructions, current file, terminal state, and other environmental data. This can be crucial for understanding the AI's behavior in an investigation.

Response Reconstruction

The AI's response is an array of heterogeneous elements. VS-CAP processes them as follows:

  1. Filter: Remove empty text elements, "thinking done" markers, and MCP server events
  2. Classify: Identify each element as text, thinking, or tool call based on the kind field
  3. Consolidate: Merge adjacent text elements into continuous blocks (VS Code may split text across multiple array entries)
  4. Render: Text → Markdown rendering; Thinking → collapsible blocks; Tools → labeled invocation cards

Response Consolidation

// Before consolidation:
[
  { type: 'text', text: 'Here is ' },
  { type: 'text', text: 'the solution:\n\n' },
  { type: 'tool', message: 'Reading file...' },
  { type: 'text', text: '```python\nprint("hello")\n```' }
]

// After consolidation:
[
  { type: 'text', text: 'Here is the solution:\n\n' },
  { type: 'tool', message: 'Reading file...' },
  { type: 'text', text: '```python\nprint("hello")\n```' }
]

Timestamp Normalization

All timestamps in VS Code's data are stored as Unix epoch milliseconds (milliseconds since January 1, 1970 UTC).

// Raw timestamp from data:
1773609980479

// VS-CAP converts to locale-aware string:
new Date(1773609980479).toLocaleString()
// → "3/15/2026, 2:06:20 PM" (varies by browser locale)

Key timestamp fields:

Forensic Methodology

Collection Best Practices

  1. Use KAPE with SYSTEM privileges: PsExec64.exe -s kape.exe --tsource C: --tdest E:\output --target VSCode_WorkspaceStorage
  2. Hash the output: Generate SHA-256 hashes of all collected files before analysis
  3. Preserve timestamps: KAPE with RecreateDirectories: true preserves file system timestamps
  4. Document the chain: Record who collected, when, from which machine, and with what tool version

Analysis Considerations

  • Chat sessions may contain sensitive code, credentials, or intellectual property
  • The renderedUserMessage reveals the full AI prompt context including system instructions
  • Tool invocations show file reads, terminal commands, and external API calls made by the AI
  • Thinking blocks reveal the AI's reasoning process, which may differ from the visible response
  • The state.vscdb database shows recently opened files and editor activity patterns
  • Agent memory files may contain plans, strategies, or accumulated knowledge from the AI

Data Integrity

VS-CAP operates as a read-only viewer. It does not modify any source files. Key integrity measures:

  • All data is loaded into JavaScript memory as immutable Uint8Array buffers
  • No data is written to disk, cookies, localStorage, sessionStorage, or IndexedDB
  • The original file bytes are preserved — parsing creates separate derived objects
  • The "Raw Files" tab allows viewing original file contents for verification
  • Export produces new JSON/CSV files from parsed data without modifying originals
✓ Verification To verify VS-CAP's parsing accuracy, compare the conversation view with the raw JSONL file in the "Raw Files" tab. The JSONL mutations should reconstruct to the exact state shown in the conversation view.
